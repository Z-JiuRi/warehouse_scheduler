## 一、架构设计与 LLM 定位

### Q1: 为什么选择"LLM 理解 + 确定性算法求解"的分层架构，而不是直接让 LLM 端到端生成路径？

**考察点：** LLM 能力边界认知、安全关键系统设计原则、架构 trade-off。

**参考答案：**

核心原因是**安全性（safety）和可验证性（verifiability）**：

1. **LLM 本质是概率模型，不适合做精确几何推理。** 路径规划要求每一步坐标严格在边界内、不穿越障碍、时间步连续递增，这些是确定性约束。LLM 生成坐标经常出现越界、跳步、穿墙等问题，无法通过形式化验证。

2. **责任分离（separation of concerns）：** LLM 擅长的是"理解用户说什么"——从"北侧通道"映射到 `corridor_north`、从"装卸区"映射到 `location_id: loading_zone`。A* 擅长的是"找到最短无碰撞路径"。各司其职是最稳健的设计。

3. **可审计性：** 当调度失败时，我们需要明确知道是"解析错了"还是"算法找不到解"。如果端到端用 LLM，失败原因是一个黑盒。

4. **成本与延迟：** 时空 A* 在 10×10 网格上运行只需 0.1ms，而 LLM 推理需要数百 ms 到数秒。把 LLM 调用次数降到最低（每个请求仅 1 次解析调用）是工程上的合理选择。

本项目严格遵循 AGENTS.md 规则：
> LLM 只负责理解和决策；路径搜索、冲突检测和验证由确定性工具完成。

**追问：** "如果换成 GPT-4 多模态能力，能否让它直接看地图图片输出路径？"
- 答：可以尝试但不应作为生产方案。多模态模型对空间推理的精确度仍不可靠，且每次推理成本远高于 A*。但可以作为辅助——例如让模型"看懂"地图并标注语义位置。

### Q2: 为什么用 LangGraph 而不是自己写一个简单的状态机？

**考察点：** 框架选型的工程判断、对 LangGraph 原理的理解。

**参考答案：**

1. **显式的图拓扑带来可维护性。** 本项目的调度流程包含 10 个处理阶段、条件分支（有冲突→重规划、无冲突→验证）、循环边（最多 3 次重试）。手写状态机容易变成意大利面条式代码。LangGraph 的 `StateGraph` 用声明式方式定义节点和边，流程一目了然：

```python
builder.add_edge("initial_plan", "conflict_check")
builder.add_conditional_edges("conflict_check", route_fn, {
    "validate": "validate_final",
    "replan": "replan_decide",
    "partial": "partial_execution",
    "infeasible": "compute_metrics",
})
```

2. **状态管理。** `GraphState` TypedDict 提供类型安全的状态传递，累积字段（replan_history、errors）使用 `operator.add` reducer 自动追加而非覆盖，避免了手动管理列表合并的 bug。

3. **可扩展性。** 后续可以零改动加入：checkpoint（中断恢复）、streaming（实时进度推送）、human-in-the-loop（在 replan_decide 前暂停等待人工审批）。

4. **与 LangChain 生态互通。** 如果需要接入更复杂的 LLM 调用链、工具调用，LangGraph 天然兼容。

---

### Q3: GraphState 中为什么有些字段用 `operator.add` reducer，有些用默认覆盖？

**考察点：** 对 LangGraph 状态 reducer 机制的理解。

**参考答案：**

`operator.add` 用于**只增不减的累积字段**：
- `replan_history`：每次重规划追加一条决策记录，不能覆盖之前的
- `errors`、`warnings`：多个节点可能各自产生诊断信息，需要汇总
- `task_results`：最终结果列表，由 compute_metrics 一次性追加

默认 `operator.setitem`（覆盖）用于**每个阶段重新设定的字段**：
- `status`：每个节点可能改变批次状态（PARSED → VALIDATED → INITIAL_PLANNED → ...）
- `current_paths`：每次重规划都会产生全新的路径集
- `retry_count`：严格递增但每次覆盖为最新值

设计原则：**凡是有多个节点可能独立产生、需要合并的信息，用 add；凡是单一事实源、越新越准确的数据，用覆盖。**

## 二、时空 A* 与路径规划算法

### Q4: 为什么必须用时空 A* (x, y, t) 而不是普通二维 A* (x, y)？

**考察点：** 多机器人路径规划的基础理论。

**参考答案：**

普通二维 A* 找到的是**空间最短路径**，但它无法表达"时间"概念，导致三个致命问题：

1. **无法处理等待。** 多机器人场景中，低优先级机器人可能需要原地等待让高优先级先通过。二维 A* 的状态只有 (x, y)，无法区分"t=5 时在 (3,3) 等待"和"t=6 时在 (3,3) 等待"。

2.  **无法表达交换冲突。** 两台机器人在相邻格子、同一时间段交换位置（A: (1,0)→(0,0)，B: (0,0)→(1,0)），二维路径看起来都没问题，但实际执行中会碰撞。时空 A* 能检测到 A 在 t 时刻从 (1,0) 到 (0,0) 的边与 B 从 (0,0) 到 (1,0) 的边交叉。

3. **无法建模动态障碍。** 临时封闭通道、其他机器人的预留路径都是时间相关的——"t=10 到 t=50 之间 (3,3) 被占用"。二维障碍集无法表达这一点。

时空 A* 的状态 `(x, y, t)` 将问题从 2D 扩展到 3D，搜索空间虽然变大，但在 10×10×200 的规模下完全可控。

**追问：** "为什么不直接用 CBS (Conflict-Based Search)？"
- 答：CBS 是完备的多机器人求解器，但对于 3-5 台机器人、10×10 地图，优先级规划+预留表已经足够高效且实现简单。CBS 的二叉树搜索在冲突多时开销更大。AGENTS.md 已将 CBS 列为未来扩展项。

### Q5: A* 的启发函数用的是曼哈顿距离，为什么？有更好的选择吗？

**考察点：** A* 启发函数设计、网格图性质。

**参考答案：**

曼哈顿距离 `h = |x - gx| + |y - gy|` 在纯网格、四方向移动（无对角线）的场景中是**最优且 admissible 的**：

1. **Admissible（可采纳）：** 曼哈顿距离永远不会高估实际代价，因为每步只能移动一个单位（上下左右），最短路径至少需要 `|Δx| + |Δy|` 步。
2. **Consistent（一致性）：** 对于四方向移动，`h(s) ≤ cost(s, s') + h(s')`，保证 A* 第一次展开到目标就是最优路径。

在当前场景下没有更好的选择：欧几里得距离在网格上不 admissible（会低估，因为不能走斜线）；对角线距离需要允许斜向移动。

如果后续加入**转向成本**或**不同地形代价**，启发函数需要相应调整权重。

### Q6: 预留表（Reservation Table）的设计要点是什么？如何处理终点持续占用？

**考察点：** 多机器人时空冲突避免机制。

**参考答案：**

预留表的核心数据结构：

```python
_vertex_reserved: Dict[Tuple[int, int, int], str]  # (x, y, t) → robot_id
_edge_reserved: Dict[Tuple[int, int, int, int, int], str]  # (x1, y1, x2, y2, t) → robot_id
```

**设计要点：**

1. **同机器人豁免：** `is_free(x, y, t, robot_id)` 中如果预留者就是自己，返回 True——机器人不会和自己冲突。
2. **边预留检测交换冲突：** 不只是检查目标格，还要检查"从 A 到 B 的移动"是否与"从 B 到 A 的移动"在同一时间步冲突。
3. **等待安全：** `is_wait_safe()` 确保原地等待时不会被其他机器人穿过的边撞到。
4. **终点持续占用填充：** 这是最容易遗漏的点。机器人 t=17 到达终点后，在 t=18, 19, ..., max_t 持续占用终点格子。在 reserve_path 时，`for t in range(last_t + 1, max_timestep + 1)` 填充终点位置。

如果没有终点填充，后到达的机器人可能在 t=18 时规划经过已被占用的终点格，导致运行时碰撞。

## 三、冲突检测与重规划

### Q7: 冲突检测系统覆盖了哪些冲突类型？各自如何检测？

**考察点：** 多机器人路径冲突的形式化定义。

**参考答案：**

四种冲突类型，按检测方式：

| 类型 | 检测方式 | 示例 |
|------|---------|------|
| **顶点冲突** | `pos_a == pos_b` 且 `t_a == t_b` | 两台机器人在 t=5 同时占据 (3,3) |
| **交换冲突** | `prev_pos_a == pos_b` 且 `prev_pos_b == pos_a` | A: (2,3)→(3,3), B: (3,3)→(2,3) |
| **起点冲突** | `t=0` 时 `pos_a == pos_b` | 两台机器人的起点相同 |
| **终点冲突** | `last_pos_a == last_pos_b` | 两台机器人分配到同一个目标格 |

核心实现细节：

1. **路径补齐（padding）：** 较短路径的机器人在到达终点后，填充终点坐标到 max_t，确保与较长路径的机器人在所有时间步都有对应位置可比。
2. **去重：** 同一冲突可能被多次检测（如顶点冲突和起点冲突重叠），通过 `(conflict_type, sorted(robot_ids), time, position)` 四元组去重。

**追问：** "交换冲突和边冲突是同一个概念吗？"
- 是。交换冲突是最常见的边冲突——两台机器人互换位置。更广义的边冲突还包括交叉但不互换了（如 A 从 (1,1)→(2,1)，B 从 (1,2)→(2,1)，同时到达同一格），这在预留表的边检测中已经覆盖。

### Q8: 三级重规划策略的具体内容是什么？为什么恰好是三级？

**考察点：** 重规划策略设计、优先级规划原理。

**参考答案：**

三级策略对应 `retry_count` 0→1→2，每次失败后升级手段：

| 级别 | retry_count | 策略 | 原理 |
|------|-------------|------|------|
| **Level 1** | 0 | **固定高优先级，重规划低优先级** | 为高优先级机器人建立预留表，低优先级机器人在预留表的约束下用 A* 重新规划（含等待和绕行） |
| **Level 2** | 1 | **扩大冲突点时间窗口** | 在冲突格子周围添加 ±2 时间窗口的顶点阻塞，强制低优先级机器人更早绕行或更晚到达 |
| **Level 3** | 2 | **局部优先级互换** | 交换冲突双方的任务优先级，打破原先的"先来先占"锁死局面 |

**为什么是 3 级而非更多？**

1. 每个级别对应不同的问题根源：优先顺序→路径空间→资源分配。3 级覆盖了主要解决方案空间。
2. 3-5 台机器人的组合空间有限，3 次重试在实践中足以解决大部分冲突。超过 3 次说明问题本身可能不可行。
3. 每次重规划都运行 A*，消耗计算资源，且增加延迟。3 是 AGENTS.md 中已确认的上限。

**追问：** "Level 3 的优先级互换如果产生新的冲突怎么办？"
- 新冲突会触发新一轮的 conflict_check → replan_decide 循环。如果 retry_count 已达 3，则走 partial_execution 路径。

## 四、LLM 应用与 Prompt Engineering

### Q9: TaskParserAgent 的 Prompt 设计有哪些精心考虑的点？

**考察点：** Prompt Engineering、结构化输出、容错设计。

**参考答案：**

`PARSE_SYSTEM_PROMPT` 的设计包含以下关键要素：

1. **严格的 JSON Schema 约束：** 要求返回 `{"tasks": [...], "constraints": [...], "parse_warnings": [...], "parse_errors": [...]}` 纯 JSON，禁止模型自由发挥。

2. **地图上下文注入：** 通过 `{locations_info}`、`{corridors_info}`、`{robots_info}` 三个占位符，将地图中的语义位置 ID、别名、入口格坐标、可用机器人 ID 注入 Prompt，让模型在已知的"词汇表"内选择，而非编造。

3. **坐标不猜测原则：** 明确规则 "start 坐标如果用户没指定，填 null，系统用机器人当前位置补全"。这从根本上避免了 LLM 编造坐标。

4. **优先级编码：** "数字越小优先级越高。用户指定则按用户，否则按出现顺序 (1, 2, 3...)"。将业务规则显式编码到 Prompt 中。

5. **错误与警告分离：** 不存在的机器人放 `parse_errors`（阻断执行），无法理解的内容放 `parse_warnings`（继续执行但记录）。这种分级处理避免了"一个拼写错误导致整批失败"。

6. **Markdown 容错：** `_extract_json()` 方法先用 `json.loads()` 解析纯 JSON，失败后用正则提取 `{...}` 块，然后再用 `json.loads()`。兼容了 DeepSeek 模型可能输出 ` ```json ... ``` ` 包裹的情况。

**追问：** "temperature 设为 0.1 的考虑是什么？"
- 任务解析是确定性的结构化提取，不需要"创意"。low temperature 提高输出格式的稳定性，减少幻觉。但在复杂自然语言表述下，0.1 而非 0 保留了最低限度的语言理解灵活性。

### Q10: LLM 解析失败后的 fallback 机制是什么？

**考察点：** 系统鲁棒性、错误处理。

**参考答案：**

四级 fallback：

1. **API 调用异常：** `try/except` 捕获网络错误、超时、鉴权失败 → 返回带 `parse_errors` 的空 `TaskBatch`。
2. **JSON 解析失败：** 先尝试 `json.loads()`，失败后用正则 `re.search(r'\{[\s\S]*\}', text)` 提取 → 再失败返回错误。
3. **Schema 校验失败：** 逐任务验证坐标边界、位置 ID 存在性、优先级合法性，失败放 `parse_errors`。
4. **结构化模式完全绕过 LLM：** `--structured` CLI 参数和 `workflow.run_structured()` API 直接接受 JSON，零 LLM 依赖，离线可用。

这种设计确保了系统在没有 API Key 或网络不通时仍然可用（结构化模式），且任何 LLM 输出异常都不会导致系统崩溃。

## 五、部分执行与调度优化

### Q11: 部分执行（Partial Execution）的搜索策略是什么？时间复杂度如何？

**考察点：** 组合优化、搜索算法设计。

**参考答案：**

策略：**贪心 + 优先级驱动移除。**

```python
sorted_tasks = sorted(all_tasks, key=lambda t: -t.priority)  # 低优先级排前面
removed_robots = set()

for task in sorted_tasks:
    if task.robot_id not in failed_robots:  # 只移除参与冲突的
        continue
    removed_robots.add(task.robot_id)
    result = plan_subset(remaining_tasks, occupied_starts=removed)
    if result is not None:
        return PARTIALLY_SUCCEEDED
return INFEASIBLE
```

**时间复杂度：** O(n² × A*)
- 最坏情况：n 台机器人全部冲突，逐个尝试移除，每次移除后重新规划剩余子集（每个机器人跑一次 A*）。
- n=3~5，每次 A* 在 10×10×200 空间上约 0.1ms，总耗时可忽略。

**优化点：**
- 按优先级排序：优先移除低优先级机器人，保留高优先级任务。
- occupied_starts：被移除的机器人持续占用起点，防止剩余机器人"借用"已被占的位置。
- 找到第一个可行子集立即返回，不继续搜索更优子集（贪心）。

**追问：** "如果 n=100 台机器人，这个策略还能用吗？"
- 不能。组合搜索会爆炸。需要更高效的子集选择（如最大独立集、约束满足问题求解）。但本项目 scope 固定在 3-5 台。

### Q12: 系统如何处理"到达终点后持续占用"这一约束？如果不处理会有什么问题？

**考察点：** 多机器人时空约束的边界条件。

**参考答案：**

**实现方式：**

1. **冲突检测中的填充：** 路径 A 在 t=17 到达终点 (9,8)，路径 B 在 t=20 到达终点 (0,8)。检测冲突时，路径 A 的 (9,8) 被填充到 t=18, 19, 20, ... max_t。如果路径 B 恰好经过 (9,8) 在 t=18，就会被检测为顶点冲突。
2. **预留表中的填充：** `reserve_path()` 中填充终点占用，确保后续规划的低优先级机器人绕过已占用的终点。
3. **目标分配去重：** `validate_and_resolve_goals` 节点中维护 `occupied_goals` 集合，同一入口格不会分配给两台机器人。

**如果不处理：**
- 机器人 A 到达装卸区后 "消失"，机器人 B 的路径穿越装卸区 → 运行时碰撞。
- 两台机器人分配到同一个入口格 → 终点冲突无法检测。
- 重规划时低优先级机器人可能规划出穿越高优先级机器人终点的路径。

## 六、系统工程与测试

### Q13: 93 个测试覆盖了哪些关键路径？测试策略的设计思路是什么？

**考察点：** 测试方法论、关键路径识别。

**参考答案：**

三层测试金字塔：

| 层 | 数量 | 覆盖 |
|----|------|------|
| **单元测试** | 80 | 每个确定性工具独立验证 |
| **集成测试** | 13 | 端到端流程 + 边界场景 |

**单元测试覆盖的关键路径：**
- A* 14 个：正常路径、障碍物、等待、动态障碍、起点/终点阻塞、越界、路径连续性、代价、最大时域
- 冲突检测 8 个：顶点、交换、起点、终点、终点填充、单机器人空路径
- 路径验证 13 个：连续性、边界、障碍、时间单调性、多机器人安全
- 预留表 10 个：顶点/边预留、路径预留、move_safe、wait_safe、swap 检测

**集成测试覆盖的边界场景：**
- 无冲突两机器人 ✓
- 三机器人全成功 ✓
- 同起点冲突 ○
- 同终点冲突 ○
- 通道封闭 ○
- 部分执行（强制冲突无法解决）○
- 路径连续性端到端验证 ✓
- 指标计算 ✓

**设计思路：**
1. 确定性工具必须有 100% 的边界覆盖——坐标越界、空输入、极限参数。
2. 集成测试使用结构化 JSON（绕过 LLM），确保测试可复现、不依赖外部 API。
3. 冲突场景刻意构造极端情况（同起点、同终点），检验重规划+部分执行的完整链路。

### Q14: 如果要在生产环境部署，这个系统还需要哪些改进？

**考察点：** 系统工程思维、从 MVP 到生产的 gap 分析。

**参考答案：**

1. **执行闭环：** 当前只输出路径，不真正下发到机器人。需要增加执行状态机（PENDING → DISPATCHED → EXECUTING → COMPLETED），接收机器人实时位置回传。

2. **在线重规划：** 当机器人偏离预定路径（如遇到意外障碍），需要 D* Lite 或 anytime 算法在线局部重规划，而非重新跑整个批次。

3. **可观测性：** 结构化日志、请求 ID 追踪、Prometheus 指标、Grafana 面板。当前指标只输出到终端。

4. **LLM 鲁棒性：** 增加 Prompt 版本管理、A/B 测试框架、解析成功率监控。当前 DeepSeek 单一模型，无 fallback 模型。

5. **完备性提升：** 优先级规划可能错过可行解。3-5 台机器人目前够用，规模扩大后应切换到 CBS/ECBS。

6. **安全性强化：** 增加路径执行前的二次确认（human-in-the-loop 或 rule-based safety check）。当前路径只经过自动验证。

7. **API 化：** 当前 CLI 模式。需要 HTTP API、鉴权、限流、幂等性保证。

## 七、综合思维题

### Q15: 如果用户说"R1 去装卸区，但别走北侧通道"，而北侧通道恰好是必经之路，系统会如何表现？

**考察点：** 系统边界行为、失败模式分析。

**参考答案：**

1. **解析阶段：** TaskParserAgent 将 "别走北侧通道" 解析为 `runtime_constraint: {constraint_type: "closed_corridor", target_id: "corridor_north"}`。

2. **障碍构建：** `build_obstacles` 节点将 corridor_north 的所有格子 (1,1) 到 (5,1) 在 t=0 到 max_t 都标记为动态障碍。

3. **初始规划：** A* 在搜索时发现从起点到装卸区的所有路径都经过 y=1 这一排（北侧通道），因为它是连接地图左右两侧最直接的横向通道。如果地图布局使得北侧通道是唯一通道，A* 遍历完 max_timestep=200 步仍找不到路径 → `FailureReason.NO_PATH`。

4. **冲突检测后：** R1 初始规划失败 → `failed_initial` 包含 R1。如果其他机器人都成功，系统进入 PARTIALLY_SUCCEEDED。

5. **部分执行：** 如果只有 R1 一台机器人，partial_execution 发现 n≤1 → 返回 INFEASIBLE。

6. **最终输出：** batch_status = `infeasible`，R1 的 failure_reason = `"no_path"`。

**关键教训：** 系统不会"勉强"给出一条经过封闭通道的路径——宁可失败也不给不安全的结果。这是安全关键系统的正确行为。

### Q16: 为什么选择优先级规划而非协商式规划（如 CNP 合同网协议）？

**考察点：** 多智能体协调模式选择。

**参考答案：**

| 维度 | 优先级规划 | 协商式规划 (CNP) |
|------|-----------|-----------------|
| 通信轮次 | 0（中心化计算） | 多轮（招标→投标→授权） |
| 完备性 | 依赖优先级顺序 | 可通过协商找到可行解 |
| 实现复杂度 | 低 | 高（需要消息协议） |
| 适用规模 | 小规模、同构机器人 | 大规模、异构机器人 |
| 本项目的匹配度 | ✅ 3-5 台同构 | ❌ 过度设计 |

在 3-5 台同构机器人、已知全局地图的场景下，中心化优先级规划是最简单高效的方案。如果未来的机器人数量增长到 50+ 且异构（不同速度/尺寸），CNP 或拍卖式分配才会显现优势。

### Q17: LangGraph 的 checkpoint 机制在这个项目中可以如何利用？

**考察点：** LangGraph 进阶特性、实际应用想象力。

**参考答案：**

虽然当前 MVP 未使用 checkpoint，但架构已为它做好准备：

1. **中断恢复：** 在 `conflict_check` 之后设置 `interrupt_after`，如果重规划 3 次仍失败，可以保存完整的图状态到 checkpoint。运维人员可以查看状态后手动调整优先级或约束，然后从断点继续执行。

2. **人工审批节点：** 在 `replan_decide` 之后插入 human-in-the-loop 节点，让人审核 AI 生成的重规划决策。如果批准则继续 `apply_replan`，如果否决则修改决策参数后再继续。

3. **时间旅行调试：** 使用 `get_state_history()` 回溯任意历史状态，复现"为什么第 2 次重规划选择了这个机器人"的决策过程。

4. **批量测试：** 对同一批任务注入不同的随机种子，用 checkpoint 保存每个分支的中间状态，对比不同策略的效果。

### Q18: 如果需要支持 100 台机器人在 1000×1000 网格上调度，当前架构的瓶颈在哪里？如何改造？

**考察点：** 系统扩展性分析、算法升级路径。

**参考答案：**

**瓶颈分析：**

| 组件 | 当前实现 | 100台×1000² 下的问题 |
|------|---------|---------------------|
| 时空 A* | 单线程、每台独立跑 | 100 台顺序跑太慢；搜索空间 1000²×T 爆炸 |
| 冲突检测 | O(n² × T) 两两对比 | 100台 = 4950 对 |
| 优先级规划 | 固定优先级顺序 | 错过可行解概率急剧上升 |
| 预留表 | 密集字典存储 (x,y,t)→id | 内存 O(n × T) |
| 部分执行 | 贪心逐个移除 | 子集数 2^100，不可行 |

**改造方案：**

1. **A* 加速：** 用 jump point search (JPS) 替代 A*（在均匀网格上剪枝）；或预计算 all-pairs shortest path + 时空维度在线查表。

2. **分层规划：** 高层用道路图（roadmap）规划宏观路径，低层用 A* 做局部避障。类似分层强化学习的思路。

3. **CBS/ECBS 替代优先级规划：** 冲突树搜索是完备的且适合中等规模；ECBS 用 focal search 进一步提升效率。

4. **预留表压缩：** 用时间区间树或 R-tree 代替密集字典，存储 (x,y,[t_start,t_end]) 区间。

5. **并行化：** A* 搜索和冲突检测天然可并行，用多进程或 GPU 加速。

6. **增量重规划：** 不重跑全量，只重规划受影响机器人（当前重规划只 replan 低优先级，已接近增量）。
